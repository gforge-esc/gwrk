# ADR-007: Single Dispatch Path & `.agents/` Deprecation

> **Status:** Proposed · **Date:** 2026-05-22
> **Decision:** All workflow dispatch (define, ship, review) flows through `WorkflowRuntime`. The `.agents/` directory is deprecated — `src/plugins/builtins/` is the canonical source for workflows, skills, rules, and review prompts.
> **Depends on:** ADR-006 (Plugin Agent Backends), F014 (Plugin System), F004 (Ship Loop)
> **Author:** David Gonzalez · **Decision Scope:** gwrk dispatch architecture, governance file authority, `.agents/` lifecycle

---

## 1. Context

gwrk's dispatch architecture has two independent paths that emerged during the F014 Plugin System rollout:

1. **Define pillar** — Commands (`specify`, `plan`, `define tasks`) were rewired to use `WorkflowRuntime.executeWorkflow()` during F014 Phase 5-6. This path resolves plugins via `PluginLoader`, loads `PROMPT.md`, injects `outputSchema`, parses JSON intent output, validates against the schema, and executes intents through `IntentEngine`.

2. **Ship pillar** — The `ShipOrchestrator` predates F014 and dispatches work via raw `dispatchToAgent()` calls with inline prompt strings. Review stages (`stageCodeReview`, `stageUatReview`) call `resolveReviewPlugin()` to get a workflow name, but then pass that name as a logging label — never loading the plugin's `PROMPT.md`.

This dual-path architecture was introduced when F014 Phase 8 (Review Plugin Layer) was partially shipped. The `ReviewPlugin` interface, `ReviewStep` schema, and `validatePhaseScope()` were built. But the 4 tasks that wire the ship orchestrator to use `WorkflowRuntime` for review dispatch (T050-T053) were cancelled. The result:

- **Review agents during `gwrk ship` receive a 10-line inline scope constraint**, not the 269-line production prompt in `gwrk-review-code.md`.
- **Every `WorkflowRuntime` improvement** (JSON intent engine, output schema validation, WRITE_FILE guards, tasks.json overwrite protection, tolerant prose mode) **is dead code during ship reviews**.
- **Review prompt improvements** made to `.agents/workflows/gwrk-review-code.md` are invisible to the ship loop, which never reads that file.

Simultaneously, `.agents/` — the pre-F014 governance directory — persists as a parallel source of truth:

| Content | `.agents/` (legacy) | Plugin system (F014) |
|---------|---------------------|----------------------|
| Workflows | 16 `.md` files | 10 builtins in `src/plugins/builtins/workflows/` |
| Skills | 11 `SKILL.md` files | Migrated to `~/.gwrk/plugins/skills/` |
| Rules | 2 governance files | **No equivalent** |
| Personas | 3 persona files | **No equivalent** |
| Review prompts | 269-line production prompt | 41-line skeleton in `builtins/reviews/` |

The F014 spec explicitly states the end goal: "ending the reliance on personal `.agents/` directories" (L48), "eliminating reliance on raw shell scripts or `.agents/workflows/` paths" (FR-L25-003), "Core workflows shipped in `builtins/workflows/` MUST be self-contained and not rely on files in `.agents/`" (TC-011).

### Forces

- **Prompt improvements are lost.** Lessons encoded in the 269-line review prompt (scope constraints, remediation format, gate-driven verdict protocol) never reach the ship loop. This has caused repeated review quality issues.
- **Silent degradation.** `resolveReviewPlugin()` catches resolution failures and falls back to a 100-line hardcoded default. The system appears to work but runs on lobotomized prompts.
- **Governance drift.** `GEMINI.md` (generated by `syncGovernance`) points to `.gwrk/rules/` which doesn't exist. `AGENTS.md` points to `.agents/rules/` which does. Neither is wrong — they're just unsynchronized.
- **Migration was designed but not completed.** `migrate.ts` exists to move `.agents/` → `~/.gwrk/plugins/`. `init.ts` already seeds from `builtins/`, not `.agents/`. The plugin loader already checks builtins first. The plumbing is in place; the switch was never thrown.

## 2. Decision

### 2.1 Single Dispatch Path: All Workflows Through `WorkflowRuntime`

The `ShipOrchestrator` review stages (`stageCodeReview`, `stageUatReview`) MUST dispatch through `WorkflowRuntime.executeWorkflow()`, the same path used by `DefineOrchestrator`.

```typescript
// BEFORE (legacy path — raw dispatch, inline prompt)
private async stageCodeReview(): Promise<StageResult> {
  const plugin = await resolveReviewPlugin(this.config.cwd);
  const scopedPrompt = `Phase ${this.config.phaseId} Code Review\n...`;
  return this.executeReviewWorkflow(plugin.codeReviewWorkflow, scopedPrompt);
  // ↑ calls dispatchWithFailback() which calls dispatchToAgent() directly
  //   never loads PROMPT.md, never parses JSON output, never validates schema
}

// AFTER (unified path — WorkflowRuntime with full plugin resolution)
private async stageCodeReview(): Promise<StageResult> {
  const plugin = await resolveReviewPlugin(this.config.cwd);
  const scopedPrompt = this.buildReviewScopeContext();  // phase tasks, done-when
  const result = await this.runtime.executeWorkflow(
    plugin.codeReviewWorkflow,
    scopedPrompt,
    { projectRoot: this.config.cwd, agent: this.config.backend }
  );
  // ↑ resolves plugin → loads PROMPT.md → injects schema → dispatches →
  //   parses JSON → validates → executes intents
  return this.interpretReviewResult(result);
}
```

**The `IMPLEMENT` stage keeps raw dispatch.** Implementing agents need native tool access (file creation, git operations, test execution). They don't return structured JSON — they modify the filesystem directly. The raw `dispatchWithFailback()` path remains correct for implementation.

**Gate-driven verdict authority is preserved.** After `WorkflowRuntime` returns the JSON result (containing the agent's `verdict` and `reopenedTasks`), the orchestrator still runs `readVerdict()` which executes gate scripts and uses their pass/fail results as the authoritative verdict. The agent's verdict is advisory. Gates are truth.

### 2.2 `.agents/` Deprecation & Removal

The `.agents/` directory is deprecated. `gwrk` MUST function correctly if `.agents/` is deleted.

**Source of Truth Hierarchy:**

| Content Type | Canonical Source | Deployed To | How |
|---|---|---|---|
| Workflows | `src/plugins/builtins/workflows/` | `~/.gwrk/plugins/workflows/` | `gwrk init` copies builtins |
| Skills | `src/plugins/builtins/skills/` | `~/.gwrk/plugins/skills/` | `gwrk init` → `seedSkills()` |
| Review plugins | `src/plugins/builtins/reviews/` | `~/.gwrk/plugins/workflows/` | `gwrk init` copies builtins |
| Rules | `src/plugins/builtins/rules/` (NEW) | `.gwrk/rules/` (project-local) | `gwrk init` seeds project |
| Personas | Inlined in PROMPT.md | — | No separate file |
| Agent adapters | `src/plugins/builtins/agents/` | Compiled into binary | `pnpm build` |

**Key invariant:** `gwrk init` is the only provisioning command. It seeds from compiled builtins. It does not read `.agents/`. A freshly initialized project has zero dependency on `.agents/`.

**`.agents/` lifecycle:**

| Phase | State |
|---|---|
| Pre-F014 | Active source of truth. IDE extensions read workflows and skills from here. |
| F014 Phase 7 (current) | Transitional. `migrate.ts` copies `.agents/` → `~/.gwrk/plugins/`. Builtins exist but `.agents/` is still the authoritative content. |
| **After this ADR** | Deprecated. Builtins are authoritative. `.agents/` may exist in older repos but is not read by gwrk. `gwrk plugin migrate` still works as a legacy cleanup tool. |
| Future | Deleted. `.agents/` does not exist in the repository. `drift-detector.ts` flags any repo that still has one. |

### 2.3 IDE Extension Context Files

IDE extensions (Gemini CLI, Claude Code, Codex CLI) read per-tool governance files:

| Tool | File | Content |
|---|---|---|
| Gemini | `GEMINI.md` | `syncGovernance()` writes `<!-- gwrk:begin -->` block |
| Claude | `CLAUDE.md` | Same pattern |
| Codex | `AGENTS.md` | Same pattern, plus per-agent `agents/` directory |
| Antigravity | `AGENTS.md` | Same file as Codex |

The `syncGovernance()` output currently points to `.gwrk/rules/` for Gemini and `.agents/rules/` for AGENTS.md. After this ADR, all references MUST point to `.gwrk/rules/`.

The `<!-- gwrk:begin -->` governance block references:
- `Rules: .gwrk/rules/`
- `Workflows: ~/.gwrk/plugins/workflows/`
- `Skills: ~/.gwrk/plugins/skills/`

No reference to `.agents/` in any governance file.

## 3. Impact Analysis

### 3.1 `resolveWorkflowPath()` in `agent.ts`

This function has a backwards-compatibility path at L342-348:

```typescript
if (workflowName.includes("/") && fs.existsSync(path.resolve(process.cwd(), workflowName))) {
  return workflowName;  // raw path like ".agents/workflows/gwrk-implement.md"
}
```

**No code change needed.** After `.agents/` deletion, this condition simply stops matching. The function falls through to `PluginLoader` resolution, which is the correct path.

### 3.2 `resolveReviewPlugin()` Hardcoded Fallback

The 100-line hardcoded fallback (L159-254 in `review-plugin.ts`) silently returns default values when plugin resolution fails. This violates the Fail Fast principle.

**Change:** Replace the `catch` with a throw. If the review plugin is not installed, `gwrk init` is the remediation — not a silent default.

### 3.3 Review `PROMPT.md` Content

The 4 review plugin builtins (`review-code-cli`, `review-code-webapp`, `review-uat-cli`, `review-uat-webapp`) currently contain 41-line skeleton prompts. These MUST be updated to carry the full production prompt content currently in `.agents/workflows/gwrk-review-code.md` (269 lines).

This is content migration, not architectural change. The PROMPT.md format and manifest.yaml schema remain unchanged.

### 3.4 Test Fixtures

7 test files reference `.agents/` in fixture setup. These update when the source moves:
- `agent.test.ts` — workflow path references
- `init.test.ts` — asserts `.agents/rules/` is NOT in output
- `integration.test.ts` — creates `.agents/rules/` in temp dir
- `dispatch.test.ts` — same
- `drift-detector.test.ts` — tests `.agents/` detection (keep as-is)

## 5. Decision Record

**Position:** Single dispatch path via `WorkflowRuntime` for all structured workflows (define, review, analyze). Raw dispatch for implementation only. `.agents/` deprecated; builtins are canonical.

**Confidence:** 9/10

**Key rationale:** The dual dispatch path was an implementation gap, not a design choice. F014's spec mandates elimination of `.agents/` reliance. The `WorkflowRuntime` exists, is tested, and is already the correct path for define workflows. Extending it to review workflows is additive — no existing define behavior changes. The `.agents/` removal is the completion of F014's stated goal, not a new architectural direction.

**Reversibility:** Low cost. `WorkflowRuntime` calls `dispatchToAgent` internally — the ship orchestrator's direct calls are a thin wrapper around the same function. If `WorkflowRuntime` introduces issues, the orchestrator can bypass it for reviews while retaining the full prompt content in PROMPT.md files. The `.agents/` deletion is also reversible via `git checkout`.

**Risk:** The primary risk is prompt regression — the review PROMPT.md files must carry all lessons from the 269-line `.agents/` version. This is mitigated by diffing the content during migration and by the existing gate-driven verdict system (gates catch prompt-quality issues mechanically).
